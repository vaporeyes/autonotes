# Research: AI Orchestrator for Obsidian Vault Management

**Date**: 2026-03-12
**Feature Branch**: `001-ai-orchestrator`

## 1. Obsidian Local REST API Surface

### Decision
Use the `coddingtonbear/obsidian-local-rest-api` plugin as the sole
integration point with Obsidian. Communicate via HTTPS on port 27124
with Bearer token authentication.

### Rationale
The plugin exposes a complete REST API covering file CRUD, directory
listing, search, command execution, and partial updates. It is the
most mature and widely-used REST API plugin for Obsidian.

### Key Findings

**Endpoints**:
- `GET/PUT/POST/PATCH/DELETE /vault/{filename}` -- file operations
- `GET /vault/{directory}/` -- directory listing (trailing slash)
- `POST /search/` and `POST /search/simple/` -- full-text search
- `GET /commands/` -- list all commands
- `POST /commands/{commandId}/` -- execute a command (returns 204)
- `GET /` -- health check (vault name, plugin version)

**PATCH mechanism**: Does NOT use JSON Patch (RFC 6902). Uses custom
headers:
- `Heading` -- targets a specific heading section (e.g., `## Tasks`)
- `Content-Insertion-Position` -- `beginning` or `end` of section
- `Heading-Boundary` -- stop at next heading of same/higher level

**Response formats**:
- Default: raw markdown (`text/markdown`)
- With `Accept: application/json`: structured JSON with parsed
  frontmatter, content, stat (ctime/mtime/size), and tags

**Authentication**: Bearer token in `Authorization` header. API key
generated by plugin on first setup. All endpoints require auth.

**HTTPS**: Self-signed cert on port 27124. Clients must disable cert
verification or pin the cert fingerprint.

### Alternatives Considered
- Direct filesystem access to vault: rejected because Obsidian may
  have files open/cached, and the REST API provides search + command
  execution that filesystem access cannot.
- Obsidian URI scheme (obsidian://): rejected because it only supports
  open/search/new, not programmatic read/write.

## 2. Python Markdown Parsing Stack

### Decision
Use `python-frontmatter` 1.1.0 + `ruamel.yaml` 0.18.x for frontmatter,
`markdown-it-py` 3.x for body parsing, and regex for Obsidian-specific
constructs (wikilinks, tags).

### Rationale
This combination provides surgical edit capability: frontmatter edits
via key-level mutation on the parsed metadata dict (body untouched),
body edits via line-number splicing using markdown-it-py token `.map`
offsets (no re-serialization). `ruamel.yaml` preserves YAML key ordering
and formatting for idempotent round-trips.

### Key Findings

| Need | Library | Version |
|------|---------|---------|
| Frontmatter parse/serialize | python-frontmatter | >=1.1.0 |
| YAML round-trip fidelity | ruamel.yaml | >=0.18.0,<0.19 |
| Markdown AST + line offsets | markdown-it-py | >=3.0.0,<4 |
| Wikilink/tag extraction | regex (stdlib) | -- |
| Body section insertion | markdown-it-py .map + string splice | -- |

**Surgical frontmatter edit pattern**:
1. `post = frontmatter.load(text)` -- splits frontmatter from body
2. Mutate `post.metadata[key]` -- body string untouched
3. `frontmatter.dumps(post)` -- re-serializes frontmatter only

**Surgical body edit pattern**:
1. Parse body with markdown-it-py to get token stream
2. Locate target heading via token type + `.map` line range
3. Splice new content at the target line in the raw string
4. No re-serialization of the full document

**Obsidian-specific regex**:
- Wikilinks: `\[\[([^\[\]|]+)(?:\|([^\[\]]+))?\]\]`
- Tags: `(?<!\S)#([a-zA-Z][a-zA-Z0-9/_-]*)`

### Alternatives Considered
- `mistune` 3.x: dict-based AST without formal spec, thin plugin story.
- `marko` 2.x: typed node objects but lower community activity.
- Operating directly on Obsidian REST API JSON response for parsing:
  rejected because the API returns minimal frontmatter parsing and no
  body structure.

## 3. FastAPI + Celery + Redis Architecture

### Decision
FastAPI for the API layer, Celery with Redis broker for background
tasks, PostgreSQL for persistent job/log storage, Redis for transient
Celery result backend. Deploy via Docker Compose.

### Rationale
This is a well-established stack for Python async APIs with background
processing. FastAPI provides automatic OpenAPI/Swagger docs (satisfying
the monitoring UI requirement). Celery handles task queue semantics
(progress tracking, deduplication, retries). The hybrid storage approach
(Redis transient + Postgres persistent) balances latency with durability.

### Key Findings

**Project structure**:
```
app/
  __init__.py
  main.py              # FastAPI app
  config.py            # pydantic-settings
  celery_app.py        # Celery singleton
  api/routes/          # FastAPI route modules
  tasks/               # Celery task definitions
  models/              # SQLAlchemy models
  schemas/             # Pydantic request/response schemas
  services/            # Business logic (called by tasks and routes)
```

**Task progress tracking**:
- Celery `self.update_state(state="PROGRESS", meta={...})` for real-time
- PostgreSQL `jobs` table for persistent history and UI queries
- `AsyncResult` for in-flight status polling

**Idempotency** (three-layer defense):
1. API-layer idempotency key in PostgreSQL `jobs` table (primary)
2. Deterministic Celery task ID derived from input hash (secondary)
3. Redis lock per resource for concurrent execution prevention

**Docker Compose services**:
- `api` -- FastAPI with uvicorn
- `worker` -- Celery worker
- `redis` -- broker + transient result backend
- `postgres` -- persistent storage
- All share same image, different `command`
- `extra_hosts: host.docker.internal:host-gateway` for Obsidian access

**host.docker.internal**: Resolves automatically on macOS Docker Desktop.
Add `extra_hosts` directive for Linux portability. Obsidian REST API at
`https://host.docker.internal:27124` from inside containers.

### Alternatives Considered
- `arq` (async Redis queue): simpler but lacks Celery's progress
  tracking, task chaining, and monitoring ecosystem (Flower).
- `dramatiq`: comparable to Celery but smaller community.
- SQLite instead of PostgreSQL: rejected because Celery's DB backend
  requires SQLAlchemy-compatible database, and PostgreSQL handles
  concurrent writes from multiple workers cleanly.

## 4. LLM Integration

### Decision
Support configurable LLM provider (Claude or OpenAI) via httpx async
client. Note content sent only during explicit AI operations. Provider
endpoint and API key configured via environment variables.

### Rationale
The user specified Claude/OpenAI as providers. Using httpx (already in
the stack for Obsidian API calls) avoids adding a separate HTTP client.
Environment-based configuration supports switching providers without
code changes. Aligns with constitution Principle III (Local-First
Privacy) by making LLM calls explicit and logged.

### Key Findings
- Use Anthropic SDK or OpenAI SDK for structured API access
- Abstract behind a provider interface so both work interchangeably
- Log every LLM call in operation log (note paths sent, tokens used)
- Risk classification for auto-apply vs approval aligns with the
  provider abstraction: the orchestrator decides risk tier, not the LLM

### Alternatives Considered
- Local LLM (Ollama/llama.cpp): viable but user specified cloud
  providers. Can be added as a future provider module per Principle IV.
- LangChain/LlamaIndex: rejected for v1 as unnecessary abstraction
  over direct API calls for our use cases (summary, suggestion, chat).
